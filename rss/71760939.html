【Web前端】vue-Router踩坑 – 太气了，必须记一笔

最近在做我的毕业设计，同时也是Yurikoto第二版。前端采用的是Quasar框架v2版本，该框架基于vue3.0。我一直以来都是前端苦手，写前端debug的时候都会血压升高捶胸顿足，而今天踩坑了vue-router，此文章仅作吐槽用。

这个坑是我在把用户页面组件连接后端的时候遇到的。

用户界面的内容使用masonry风格进行排版，为了较好的动画效果与兼容性（实际上没有找到符合需求的解决方案），masonry排版使用纯js实现，这也带来了各种各样的问题，如重复加载等，我也设置了了一系列的加载、滚轮锁来试图解决问题，这也使得整个排版代码写得极为复杂，换句话说，写的跟坨屎一样。

所用用户的界面都通过同一个组件User.vue实现，路由规则设置为/user/:id来对不同用户进行区分。于是问题就出现了，当从一个用户页面跳转到另一个用户页面（如从/user/1跳转到user/2）时，界面组件并不会重新mount。这导致了一系列的问题，比如访问到的新用户的内容会在原有内容的后面继续加载，出现了一个用户的界面会刷出多个用户的内容的尴尬场面。

我最开始试图使用一个简单并且直觉的解决方案，当检测到router参数改变时，清空内容列表（内容列表组件化封装到ContentMasonry.vue中）。于是我给ContentMasonry写了一个empty方法，然后在用户界面组件用过ref调用：

empty() {
    if (this.loadingFlag) {
        this.loadingLock = true;
    }
    for (var i = 0; i < this.columnHeight.length; i++) {
        this.columnItems[i] = [];
        this.columnHeight[1] = 0;
    }
    this.itemsCached = [];
},

这个尝试看似解决了问题，但当我测试稍微极端一点的情况，比如极快的手速、极慢的接口加载速度时，还是会出现之前的尴尬场面。我也尝试了很多解决方法，比如将内容加载函数写成synchronous函数，设置各种promise, flag，都没有完美解决问题。一个小小的内容列表，搞了我两个多小时，人差点破防。

我冷静下来思考了一下这个问题，明明我访问的path不同，只不过实现的组件相同，怎么就不mount了呢？这是为什么呢？有没有办法让它mount呢？

我找到了这个帖子。一个老哥写道：

Vue Router does not notice any change if the same component is being addressed. With the key, any change to the path will trigger a reload of the component with the new data.

简直醍醐灌顶，我光速把MainLayout代码改成：

<q-page-container class="page-wrapper">
    <router-view :key="$route.fullPath" />
</q-page-container>

很棒，通过console.log的debug大法，确实在切换用户界面时，组件都会mount一遍，问题这才完美解决。

前端开发真的令人血压飙升，说白了，还是代码写少了。作这个项目之前，我根本没有系统地学习vue-router，有事就谷歌，这下尴尬了。前置技术的学习真的很重要！
