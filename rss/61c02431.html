如果重返大一学计算机，你会做什么？

罗宸的回答

学 Haskell 编程语言。


我开始接触 Haskell 是 2015 年了，感觉浪费了 6 年脑力最强的时光，在迷雾里挣扎和探索。


在C++的迷雾里:

曾经思索，为什么二分查找一定得手写，明明是在写模板代码，却不能抽象为工具库？周围的ACM大牛们甚至劝诫我说∶这种东西必须得很熟练的，因为每次题目都会出现一些变化，你不可能把它抽象成库。

还有 bfs/dfs/dp 等等。

为什么 STL 的 binary_search 一定要用于容器，怎么把函数伪装成容器然后去使用 binary_search，为什么我们不能让这一切更加通用？

为什么打印个嵌套的结构这么麻烦，以及比较，以及哈希，序列化。。。

为什么 new 和 delete 要手写，不能让机器自动帮咱处理？这个问题复杂在哪？

为什么有这么多情况编译器都不敢优化，只要存在个函数调用就连常量折叠都不敢做，编译器在担心啥？ 是可能的副作用么？

为什么不能把所有东西都表达成纯函数，为什么我们一定要依赖副作用？


在Java的迷雾里:

为什么别人的库提供的类没实现 可比较，我就不能给它提供个实现呢？

为什么我不能给基本类型实现 可比较？

为什么我不能直接告诉编译器我这个类是个数据类，一定要用lombok之类的在运行时去创建一些方法（而这个过程所依赖的信息其实早在代码运行前明明就已经确定下来了）？


在Javascript的迷雾里:

当我需要一个异步返回值的时候，为什么一定要用传一个callback这种如此不直观的方式？

为什么编写个手势识别的程序要搞这么多 callback，这面条一样的callback要怎么维护？

为什么我可以很快写出能运行的程序，却很难确保它在各种情况下都能正常工作？


直到遇到 Haskell，才知道原来可以用 higher order function 来对算法进行抽象。

原来这些算法都可以实现成库函数，你只需要接受函数可以被当作参数而已，竟如此简单！

原来可以用 type class 来给特定类型实现特定性质，而且这个实现可以在任何地方提供，不一定要在类型定义的地方。

原来 type class 还可以自动导出实现，甚至还能进行推理，比如我们可以说，如果 a 可比较，那么 a 的列表就可比较: Ord a => Ord (List a) 。

原来可以用一个容器类型（IO），把有副作用的过程和纯计算区分开，正如我们在处理异步结果时一样。

原来用两个运算符（sum/product）就能简单地表达出所有的数据（Algebraic Data Type)。

原来把异步结果抽象成一个容器类型（Promise），就可以避免callback。

原来像 IO 和 Promise 这样的容器类型，有非常多的相似行为，我们甚至可以对这些容器类型的相似性进行建模（Monad）。

原来我们可以对事件进行编程（Reactive Programming），甚至可以对一系列事件和 变化着的值进行编程 （FRP）

原来只需要通过把类型写对，我们就能获得十分可靠的程序。

原来我们甚至可以用类型来编码程序的一些性质，从而保证程序的正确性。

原来类型系统甚至可以用来做数学证明！


是学习 Haskell，才让我得以窥见计算机科学的美妙之处。

所以，如果能重返大一，我一定 ~~先梭哈比特币 ，然后用下半辈子来~~ 学 Haskell （逃
